# 同类项目对比

提示词优化相关项目，常见有四种路线：

1. 研究 / PromptOps 框架
2. 完整产品或平台
3. Claude Code / Hook / Slash Command 工具
4. 单用途 Skill / Agent Prompt 包

这个项目刻意选的是第 4 种，而且继续保持单一职责。

## 为什么不做成“大而全”

因为很多用户真正需要的不是一个庞大的 PromptOps 平台，而是一个可以在聊天里顺手调用的小能力：

- 收到一段原始需求
- 只做提示词优化
- 直接给终版提示词
- 不顺手执行任务本身
- 不把用户拖进复杂配置、框架或工作流里

这也是为什么本项目坚持：

- 单一职责
- 默认直接输出
- 不过度工程化
- 触发词尽量短
- `host-router` 也只做一轮路由，不持久占用后续对话

## 和常见路线的区别

| 路线 | 代表形态 | 更适合什么场景 | 不适合什么场景 | 本项目的区别 |
|---|---|---|---|---|
| 研究 / PromptOps 框架 | 数据集、评测、自动迭代 prompt | 团队级系统优化、离线评测、模型迁移 | 聊天里随手来一句就优化 | 本项目不是评测框架，而是即时可调用的小能力 |
| 完整产品 / 平台 | Web、桌面端、浏览器插件、MCP 平台 | 想做多模型、多模式、多入口的提示词产品 | 想要最轻量、最低前摇的调用方式 | 本项目更轻，不追求产品化面板和复杂设置 |
| Claude Code Hook / Slash Command | `/command`、hook、工作区命令 | 主要在 Claude Code 内使用 prompt 优化 | 想跨平台复用 | 本项目保持 Claude 可用，但不绑定单一宿主 |
| 多功能 Prompt 助手 | 附带追问、联网、深度模式、研究模式 | 想把提示词优化和研究/解释合在一起 | 只想拿终版提示词的人 | 本项目默认不解释、不执行；是否联网由 Skill 按需自行判断 |
| 本项目 | 单用途 Skill / Agent Prompt 包 | 想把模糊需求、已有 prompt、system prompt、agent 指令快速整理成终版提示词 | 想做系统化 PromptOps 或重产品平台 | 核心价值是“短 trigger + 单用途 + 跨平台 + 明确边界” |

## 选择建议

如果你的目标是：

- 想研究 prompt 为什么好坏不同
  - 选框架型或教学型项目
- 想做一个完整产品或 SaaS
  - 选平台型项目
- 主要在 Claude Code 里用
  - 选 hook / slash command 型项目
- 想在多个 AI 宿主里都能顺手调起一个“只负责优化提示词”的能力
  - 选本项目这种单用途 Skill

## 为什么继续保持单一职责

把“提示词优化器”做得更像一个工具，而不是更像一个大脑，长期反而更稳：

- 用户更容易理解它到底做什么
- 触发词可以更短
- 误触发边界更容易控制
- `host-router` 更容易做成一次 trigger 一次处理
- 文档、测试、接入都更容易维护

这也是为什么本项目目前不会主动扩成：

- 通用联网研究助手（但优化提示词时如需了解背景，会自行联网搜索）
- 自动执行任务的代理
- 多轮澄清型访谈机器人
- 大而全 Prompt 平台

这些方向都能做，但不应该混进“提示词优化器”这个核心定义里。
